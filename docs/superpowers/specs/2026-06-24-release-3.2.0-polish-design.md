# Release 3.2.0 — Polish & Ship Design

**Date:** 2026-06-24
**Type:** Polish + release-hygiene (two focused changes, one release)
**Status:** Approved (design) — pending spec review

## Context

A discovery pass against `origin/master` (not the stale local worktree) established the
real release state:

- **Shipped:** v3.0.0 (AI chat companion + AI inline autocomplete), v3.1.0 (catalog
  warm-up, multi-connection split view, chart view, clipboard copy).
- **Genuinely unreleased** (on `origin/master`, after the v3.1.0 tag): **query parameters
  `{{name}}` (#41)** and the `formatBytes` consolidation refactor (#40).

So the next release is a focused **3.2.0** whose single user-facing headline is query
parameters. The query-parameter implementation (`extractParams`, `substituteParams`,
`ParamsPanel`, `queryStore` wiring) was audited and is well-built: pure tested helpers,
comment/string-stripping, `reconcileParams` preserving `{type,value}` by name, and
SQL-injection-safe `text` escaping. The polish surface is small.

This spec covers the two changes worth doing before tagging 3.2.0.

> **Implementation base:** this work MUST be branched from `origin/master` (which contains
> #40/#41). The local master lineage is stale (PRs are squash-merged), so a branch cut from
> local master will be missing the query-parameter files. Branch from `origin/master`.

## Part 1 — Changelog reconciliation (release hygiene)

### Problem

`CHANGELOG.md` is out of sync with the published GitHub releases:

- It has **no `## [3.0.0]` or `## [3.1.0]` sections**.
- Its `## [Unreleased]` block still lists AI chat, AI inline autocomplete, multi-connection
  split view, chart view, and catalog warm-up as unreleased — even though they shipped in
  v3.0.0 / v3.1.0.

Because the in-app update notifier surfaces these notes, the changelog visibly diverges from
what users actually received.

### Approach

The local git tags are unreliable (the `v3.1.0` tag does not contain the split-view commit it
announced), so the **GitHub release notes are authoritative**, not commit→tag mapping.

1. **Add `## [3.0.0] - 2026-06-19`** — AI inline autocomplete + AI chat companion, copied from
   the v3.0.0 GitHub release notes.
2. **Add `## [3.1.0] - 2026-06-22`** — catalog warm-up, multi-connection split view, chart
   view, clipboard copy, plus the "editor split now spans two independent connections" Changed
   note, copied from the v3.1.0 GitHub release notes.
3. **Shrink `## [Unreleased]`** to only what was never announced to users:
   - **Added:** query parameters (`{{name}}`).
   - **Changed/Fixed:** the internal refactor notes (shared `queryRuntime`, `ResultsTable` /
     `TitleBar` decompositions, `formatBytes` consolidation, the column-resize fix). These are
     user-invisible; filing them under the upcoming 3.2.0 is honest and keeps the user-facing
     narrative aligned with the published releases.

### Judgment call (documented)

The internal refactors physically shipped inside the 3.1.0 binary but were never announced in
its release notes. Rather than retroactively rewrite the published 3.0.0/3.1.0 narrative, we
keep those two sections matching their GitHub notes verbatim and let the (invisible) internal
notes ride under Unreleased → 3.2.0. This optimizes for "the changelog matches what users were
told," which is the property the update notifier depends on.

### Acceptance

- `CHANGELOG.md` has `[3.0.0]`, `[3.1.0]`, and a trimmed `[Unreleased]`.
- The `[3.0.0]` / `[3.1.0]` Added bullets match the respective GitHub release notes.
- No already-shipped feature remains under `[Unreleased]`.

## Part 2 — Param validation at the input

### Problem

When a required `{{name}}` value is missing/invalid and the user hits **Run**, the failure
surfaces as a red error in the **results panel** ("Fill in {{foo}} before running.") — far from
the input that needs fixing. Validation logic also lives only inside `substituteParams`
(`renderValue`), so any UI-side check would risk drifting from the executor.

### Approach

**Single source of truth for param validity** (avoids the drift class we just removed by
consolidating `formatBytes`):

1. **New pure helper `src/renderer/src/lib/validateParams.ts`:**
   - `validateParam(p: QueryParam): string | null` — encodes the existing rules:
     - `text` / `number` / `boolean` require a non-empty, type-valid value;
     - `number` must be finite; `boolean` must be `true`/`false`;
     - `raw` may be empty.
     - The returned message matches today's `substituteParams` messages
       (`Fill in {{name}} before running.`, `{{name}} is not a valid number.`,
       `{{name}} must be true or false.`).
   - `validateParams(params: QueryParam[]): { name: string; message: string }[]` — all errors.
   - Unit-tested (the file is under the `lib/**` coverage gate).

2. **Refactor `substituteParams`** to call `validateParam` internally instead of its private
   `renderValue` validation, so the executor and the UI share one ruleset. The rendered-value
   production (quoting/escaping) stays in `substituteParams`; only the *validation* branch is
   delegated. Existing `substituteParams.test.ts` must stay green.

3. **`ParamsPanel`** renders per-row errors:
   - Accepts an `errors: Record<string, string>` prop (name → message) supplied by `EditorPane`.
   - An invalid row's input gets an error ring (`ring-1 ring-app-err` / border treatment
     consistent with existing focus rings) plus a small `text-app-err` message line beneath it.
   - Errors are shown only after a Run attempt (see below) — not while mid-typing — to avoid
     nagging on every keystroke.

4. **`EditorPane` guards Run (and Explain) — "block + highlight":**
   - Local state `showParamErrors: boolean` (per pane), reset when SQL/params change to a valid
     state.
   - `handleRun`: compute `validateParams(params)`. If non-empty, set `showParamErrors = true`,
     focus the first invalid input, and **return without executing** — the results panel is never
     touched. If empty, run as today.
   - Same guard in the Explain handler.
   - Run stays clickable (responsive); the click is what surfaces the highlight.
   - `ParamsPanel` receives `errors = showParamErrors ? validateParams(params) keyed by name : {}`.

5. **Remove the now-dead results-panel path:** `queryStore.runQuery` / `explainQuery` currently
   set `error: sub.error` when substitution fails. With the EditorPane guard, invalid params
   never reach the store. Keep the store's `substituteParams` error-guard as a defensive
   backstop (it can no longer be hit from the UI), OR drop it — decide during implementation; the
   acceptance below only requires that the *normal* invalid-param flow shows the error at the
   input, not the results panel.

### Out of scope

- The `raw`-param "inserted verbatim" caution treatment (finding #3) — deferred.
- Boolean-default and limit-guard-on-pre-substitution-SQL nuances — deferred (not release
  blockers).

### Acceptance

- `validateParams.ts` exists, is pure, and is unit-tested; `validateParam` messages match the
  pre-existing `substituteParams` messages.
- `substituteParams` delegates validation to `validateParam` (single ruleset); its existing
  tests stay green.
- Hitting **Run** with a missing/invalid param highlights the offending `ParamsPanel` input
  (error ring + message), focuses the first one, and does **not** execute or populate the
  results panel with the param error.
- Filling all params and hitting Run executes normally.
- `just ci` green; coverage gate holds.

## Release steps (after both parts land)

1. Finalize `CHANGELOG.md` `[Unreleased]` → `## [3.2.0] - <date>`.
2. Bump `package.json` version to `3.2.0`.
3. Update `README.md` if query-parameter docs need a touch (verify the #41 README addition is
   accurate).
4. `just ci` green.
5. Tag + release per the existing release flow.
